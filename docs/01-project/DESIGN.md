# 汉尼尔斯洞察技能（hannilce-insight-skill）设计文档

> 版本：1.0\
> 状态：设计冻结，进入实现阶段

***

## 1. 项目背景

MBTI 经过历史的验证，已被证明具有实际的应用价值。它本身不应该被简化为一套 **无聊、落俗套的机械问答**。在这个 AI 时代，我们必须要有所创新，使得这项评估工作焕发出新的魅力——**让整个过程自然流动、有对话感、有洞察力，就像一场值得回味的深度访谈，而不是一次应付了事的填表**。

MBTI 本身不是枯燥的。枯燥的只是传统的问卷形式。

在 AI 时代，我们有能力让性格评估变成**一次有温度的对话——用户不再是被测量的对象，而是被理解的个体。每一次提问都是一次共情，每一次回答都是一次自我发现。**

汉尼尔斯（hannilce-insight） 追求的正是这种转变：**用读心取代填心，用对话唤醒自知**。

***

## 2. 核心价值

| 价值点      | 说明                          |
| -------- | --------------------------- |
| **保真**   | 不直接问 MBTI 问题，通过话题引导让答案自然浮现  |
| **可解释**  | 每轮对话有 token 量、质量分记录，推断过程可追溯 |
| **自适应**  | 质量下降时封存话题，避免引导用户走向“正确答案”    |
| **持续优化** | 置信度模型独立迭代，数据越多越准确           |

## 4. 用户画像

- **存储字段**：姓名 + 性别 + 出生年月（YYYYMM）+ 职业
- **隐私策略**：仅存 YYYYMM（可计算年龄区间），无需脱敏；数据本地 SQLite 存储，不外传。
- **用途**：首次触发时可选询问，后续话题生成和评分均作为上下文输入。

***

## 5. 话题生成策略

### 5.1 设计选择

- **完全依赖 LLM 生成问题**，不维护静态话题池。
- 为防止幻觉，引入 **话题源种子**（从采集的新闻库中按用户画像筛选摘要），作为生成问题的“锚点”。

### 5.2 生成流程

1. 系统根据用户画像（职业/年龄/性别）从 `scraped_topics.json` 中筛选 1\~2 条相关新闻摘要。
2. 构建 Prompt（含画像、历史对话、目标维度、筛选出的种子素材），调用 LLM 生成问题。
3. 若种子素材不足（如采集失败），则允许 LLM 自由发挥，但明确禁止引用真实具体的事件或人物。

### 5.3 维度重复控制

- 系统维护每个维度的置信度分数。
- 当某个维度置信度超过阈值（如 0.7）时，后续轮次不再优先生成该维度的问题，除非用户主动提及。

### 5.4 温度设置

- 生成 MBTI 问题：温度 **0.4**（范围 0.2～0.5）
- 生成报告摘要：温度 **0.3**

### 5.5 新闻种子 seed\_data 采集与缓存策略

用户触发 → 检查缓存 → 新鲜则用，否则采集 → 采集失败则回退到本地缓存（若存在）→ 仍失败则纯 LLM 生成（零资料）

## 6. 评分与置信度

### 6.1 混合评分架构

| 评分项         | 方式                        | 权重  |
| ----------- | ------------------------- | --- |
| token 量     | 代码硬规则（100\~500 tokens 为佳） | 30% |
| 重复率 & 矛盾检测  | 代码计算（无外部模型依赖）             | 30% |
| 语义丰富度 & 真诚度 | LLM 评分（温度 0.2，三次取中位数）     | 40% |

### 6.2 LLM 评分防幻觉设计

**输入约束**（Prompt 模板）：

```text
你是 MBTI 对话质量评估员。请对用户回答进行三项打分（0-10分）：
1. 语义丰富度（是否表达了观点、理由、感受）
2. 一致性（是否与历史回答矛盾？若矛盾则低分）
3. 真诚度（是否像真实回答，而非敷衍或套话）

规则：
- 如果用户回答少于 10 个字，语义丰富度自动为 0。
- 如果用户回答包含明显重复上一轮的内容，一致性分 ≤ 3。
- 不要因为回答符合某种人格就刻意打高分或低分。
- 输出格式：{"semantic": 7, "consistency": 8, "authenticity": 6}

只输出 JSON。
```

**输出约束**：强制 JSON，无额外解释文本。\
**后处理校验**（代码层）：

- 范围裁剪（0～10）
- 与历史滑动平均差异过大时降权
- 异常低分且回答较长时标记“疑似幻觉”

### 6.3 三次评分取中位数

- 对同一段回答调用 LLM 评分 3 次（温度 0.2），取中位数作为最终语义分。
- 若三次中有一次明显偏离（离差 > 3 分），剔除异常值后取平均。
- **消耗评估**：3 次评分 token 成本可接受（换取高鲁棒性）。

### 6.4 置信度更新

每轮综合评分 = token量分×0.3 + 重复/矛盾分×0.3 + 语义分×0.4\
会话级置信度 = 滑动平均（最近 5 轮综合评分）\
结束条件：对话轮次 ≥ 8 且置信度 ≥ 0.7 且四维均有数据覆盖。

## 7. 长对话接管与会话恢复

### 7.1 状态保持

- Hermes/OpenClaw 均支持 session 持久化（本地 SQLite）。
- 用户标识：用户姓名 + 出生年月 YYYYMM
- 会话标识：用户姓名 + 首次触发时间戳哈希。
- 每次调用 `handle_trigger` 或 `handle_response` 都会读取/更新数据库中的会话记录。

### 7.2 中断恢复

- 技术层支持：通过 `hermes --resume <session_id>` 或自动根据用户姓名唤醒。
- **外部依赖问题**：飞书等平台 token 熔断会阻断通信信道，即使本地会话完整也无法恢复。建议将 Agent 部署在可自主控制信道的环境。

### 7.3 主动释放

- 当分析完成（`should_archive=true`）或质量持续下降触发封存时，技能输出主动终止循环并暂缓出报告，将控制权交还主 Agent。

## 8. Hermes Skill 封装规范

### 8.1 技能文件结构

```
hannilce-insight-skill/
├── skill/
│   └── SKILL.md            # 元数据 + 自然语言描述
├── mbti/                   # Python 模块
│   ├── __init__.py
│   ├── insight_skill.py    # 主入口（InsightSkill 类）
│   ├── session_manager.py
│   ├── topic_generator.py
│   ├── quality_controller.py
│   ├── models.py
│   └── db.py
├── seed_data/              # 采集 新闻/职业特征 相关话题
│   └── scraped_topics.json
├── requirements.txt        # numpy, pydantic
└── README.md
```

### 8.2 SKILL.md 关键字段

```yaml
---
name: hannilce-insight
label: 汉尼尔斯读心术
description: 基于多轮对话的 MBTI 性格分析技能
triggers:
  - /MBTI
commands:
  - name: /MBTI
    description: 开始 MBTI 分析，用法：/MBTI <姓名>
handler:
  command: "cd ${SKILL_DIR} && PYTHONPATH=${SKILL_DIR} python mbti/insight_skill.py"
---
```

### 8.3 与 OpenClaw 对齐

- 同一 `SKILL.md` + Python 脚本可直接在 OpenClaw 中安装，无需修改。
- 双方均兼容 Agent Skills 开放标准。
